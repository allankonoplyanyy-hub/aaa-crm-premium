## 4. Способ хранения данных (подтверждение)

- **PostgreSQL (Neon) — используется** как основное хранилище (`lib/server/pg-repo.ts`).
  Все данные переживают рестарты сервера и redeploy.
- **In-memory — только fallback** без `DATABASE_URL` (локальное демо).
- **localStorage — НЕ используется.**
- **Сессии** — серверные (таблица `sessions`), cookie содержит только случайный
  256-битный токен. HttpOnly, Secure (prod), SameSite=Lax, TTL 7 дней.
- **Пароли** — scrypt (N=16384, r=8, p=1); сидовые аккаунты по `DEMO_PASSWORD`.
- **Аудит** — таблица `audit_log`, пишутся все мутации и вход/выход.

## 5. Безопасность (реализовано в этой ревизии)

1. scrypt-хеширование паролей + timing-safe сравнение + dummy-verify против перебора email.
2. Серверные сессии в БД с TTL и sliding-продлением; logout инвалидирует запись.
3. CSRF: double-submit cookie + `x-csrf-token` на всех мутациях.
4. Rate limiting входа/регистрации (sliding window, per-IP+email).
5. zod-валидация payload во всех мутирующих роутах (400 с деталями).
6. Интеграционный API v1: HMAC-SHA256 + окно 5 минут + идемпотентность по `event_id`.
7. Audit log всех мутаций; structured JSON-логирование (`lib/server/log.ts`).
8. `/api/health` — readiness с проверкой БД.

## 6. Оставшиеся риски

1. Rate limiting per-instance (in-memory) — для мультиинстансного продакшена нужен Redis.
2. Общий `INTEGRATION_WEBHOOK_SECRET` на инсталляцию — per-tenant секреты в бэклоге.
3. Сидовые аккаунты по общему `DEMO_PASSWORD` — приемлемо для демо, отключить перед реальным запуском (или сменить env).
4. Нет E2E-тестов — UI-регрессии ловятся вручную.
5. Нет пагинации — при росте данных потребуется серверная фильтрация.

## 7. Вердикт

Проект технически готов к пилотным продажам: PostgreSQL-персистентность,
настоящая аутентификация (scrypt + серверные сессии), CSRF, rate limiting,
audit log, валидация и интеграционный API с HMAC. Оставшиеся пункты
(см. CODEX_HANDOFF.md, раздел 11) — production-доработки, не блокирующие пилот.
