# Backup & Restore — AAA CRM Premium

База данных: **Neon PostgreSQL** (serverless Postgres). Все данные приложения
(тенанты, пользователи, сессии, сделки, контакты, диалоги, звонки, аудит)
хранятся в одной базе, схема — `db/migrations/`.

## 1. Что делает Neon автоматически

- **Point-in-Time Restore (PITR).** Neon хранит историю WAL и позволяет
  восстановить состояние базы на любой момент внутри окна хранения
  (по умолчанию для Free-плана — 24 часа, на платных планах — до 30 дней).
  Настройка: Neon Console → Project → Settings → History retention.
- **Branching.** Восстановление выполняется созданием ветки базы от точки
  во времени (`restore branch`) — production-ветка при этом не трогается,
  что позволяет сначала проверить данные, а затем переключить приложение.

## 2. Ручной логический бэкап (pg_dump)

Рекомендуется для регулярных внешних копий (например, до крупных миграций):

```bash
# полный дамп (schema + data), сжатый custom-формат
pg_dump "$DATABASE_URL" --format=custom --file=backup_$(date +%Y%m%d_%H%M%S).dump

# только данные / только схема
pg_dump "$DATABASE_URL" --data-only  --format=custom --file=data.dump
pg_dump "$DATABASE_URL" --schema-only --file=schema.sql
```

Хранить дампы вне Neon (S3/GCS/локальный сейф), шифровать при передаче.

## 3. Восстановление

### 3.1 Из логического дампа (pg_restore)

```bash
# в чистую базу (создать новую ветку/базу в Neon, взять её connection string)
pg_restore --dbname="$NEW_DATABASE_URL" --clean --if-exists --no-owner backup.dump
```

После восстановления переключить `DATABASE_URL` в env проекта (Vercel →
Settings → Environment Variables, либо v0 → Vars) и сделать redeploy.

### 3.2 Point-in-Time Restore (Neon)

1. Neon Console → Branches → Restore.
2. Выбрать ветку-источник (production) и timestamp до инцидента.
3. Neon создаст восстановленную ветку; проверить данные запросами.
4. Переключить `DATABASE_URL` приложения на восстановленную ветку
   (или сделать её primary), redeploy.

## 4. Миграции при восстановлении

Схема управляется файлами `db/migrations/*.sql` и раннером
`node scripts/migrate.mjs` (идемпотентен: применённые версии учитываются
в таблице `schema_migrations`). После восстановления из старого дампа —
просто запустить раннер, он докатит недостающие миграции.

## 5. Что НЕ нужно бэкапить

- Сессии (`sessions`) — эфемерны, пользователи просто войдут заново.
- Rate-limit-состояние — in-memory, не хранится в БД.
- Кэш идемпотентности (`integration_events`) — желательно сохранять
  (защита от повторной доставки вебхуков), но потеря не критична:
  интеграции переотправят события с теми же `event_id`.

## 6. Регламент (рекомендация для эксплуатации)

| Действие | Периодичность |
| -------- | ------------- |
| Проверка окна PITR в Neon | при онбординге + при смене плана |
| pg_dump во внешнее хранилище | еженедельно + перед каждой миграцией |
| Тестовое восстановление в ветку | ежеквартально |
| Ротация `INTEGRATION_WEBHOOK_SECRET` | при компрометации; желательно раз в 90 дней |
